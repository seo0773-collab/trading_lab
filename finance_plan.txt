
# trading_lab 재무제표 민감도 전략 개발 계획표

## 1. 작업 목표

`trading_lab` 레포 안에서 새로운 재무제표 기반 전략을 설계한다.

전략의 목적은 특정 종목의 과거 주가가 재무제표 변화에 어떻게 반응해왔는지 학습하고, 이후 새로운 재무제표 변화가 발생했을 때 향후 주가 반응을 추정하는 것이다.

핵심 관점은 다음과 같다.

```text
이 종목은 어떤 재무 팩터 변화에 민감한가?
현재 재무제표 변화는 과거 반응 패턴상 주가에 어느 정도 영향을 줄 수 있는가?
그 반응이 현재 밸류에이션과 시장 환경에서도 유효한가?
```

## 3. 전략 개념

이 전략은 단순히 재무제표가 좋은 회사를 고르는 방식이 아니다.

중요한 것은 재무제표의 절대 수준보다 **변화와 반응**이다.

예를 들어 다음과 같은 질문에 답하는 전략이어야 한다.

```text
매출 성장률이 개선됐을 때 이 종목은 과거에 얼마나 올랐는가?
영업이익률이 악화됐을 때 주가는 얼마나 민감하게 반응했는가?
ROE가 개선돼도 이미 PER이 높으면 주가 반응이 줄어드는가?
영업현금흐름 개선은 단기보다 중기 주가에 더 잘 반영되는가?
```

최종적으로 전략은 아래 흐름을 따른다.

```text
과거 재무 상태
→ 다음 재무제표 변화
→ 발표 이후 주가 반응
→ 종목별 민감도 학습
→ 현재 재무 변화에 적용
→ 예상 수익률 산출
```

## 4. 사용할 기본 팩터

우선 아래 재무 팩터를 중심으로 설계한다.

| 구분    | 팩터               |
| ----- | ---------------- |
| 성장성   | 매출 성장률           |
| 수익성   | 영업이익, 순이익, ROE   |
| 현금흐름  | 영업현금흐름           |
| 안정성   | 부채총계, 자본총계, 부채비율 |
| 밸류에이션 | PER, PBR, PSR    |
| 주주환원  | 배당               |
| 기대치   | EPS 전망           |

단, PER, PBR, PSR은 주가가 포함된 지표이므로 단순 절대값보다 다음과 같은 상대 지표를 우선한다.

```text
과거 평균 대비 현재 PER
섹터 평균 대비 현재 PER
과거 평균 대비 현재 PBR
섹터 평균 대비 현재 PBR
과거 평균 대비 현재 PSR
섹터 평균 대비 현재 PSR
```

EPS 전망은 데이터 안정성이 낮을 수 있으므로 선택 팩터로 둔다.

## 5. 데이터 사용 원칙

가장 중요한 원칙은 **미래 데이터 누수 방지**다.

재무제표의 기준일과 실제 사용 가능일은 반드시 분리한다.

예를 들어 1분기 재무제표 기준일이 3월 31일이어도, 그 정보를 3월 31일에 이미 알고 있었다고 처리하면 안 된다.

기본 원칙은 다음과 같다.

```text
재무제표 기준일 ≠ 투자자가 정보를 알 수 있는 날
```

발표일 정보가 있으면 발표일을 사용한다.

발표일 정보가 없으면 보수적으로 다음 기준을 적용한다.

```text
분기보고서: 분기 종료 후 45일 뒤부터 사용 가능
사업보고서: 결산 종료 후 90일 뒤부터 사용 가능
```

백테스트에서는 해당 시점에 실제로 알 수 있었던 정보만 사용해야 한다.

## 6. Initialize 구간 설계

전략 초반에는 바로 매매하지 않는다.

먼저 initialize 구간을 두고, 해당 종목이 과거에 어떤 재무 변화에 민감했는지 학습한다.

기본 설계는 다음과 같다.

```text
초기 12~16개 분기 이상은 학습 전용 구간으로 사용
이 기간 동안 매매 신호를 내지 않음
각 분기별 재무 상태와 이후 주가 반응을 연결
종목별 민감도를 추정
```

initialize 구간에서 확인해야 할 내용은 다음과 같다.

| 항목        | 목적                      |
| --------- | ----------------------- |
| 재무 팩터 변화량 | 직전 분기 대비 변화 확인          |
| 발표 후 수익률  | 실제 주가 반응 확인             |
| 팩터별 민감도   | 어떤 요인에 주가가 반응했는지 확인     |
| 밸류에이션 상태  | 고평가/저평가 상태에 따른 반응 차이 확인 |
| 섹터 환경     | 업종 전체 흐름의 영향 확인         |

---

## 7. Rolling 업데이트 구조

initialize 이후에는 고정된 민감도만 사용하지 말고 rolling 방식으로 업데이트한다.

기본 구조는 다음과 같다.

```text
최근 12~16개 분기를 기준으로 민감도 재학습
새 재무제표 발표 시 예상 수익률 계산
실제 주가 반응 확인
다음 분기부터 민감도 업데이트
```

이 방식은 종목의 성격이 시간이 지나며 바뀌는 문제를 완화하기 위한 것이다.

예를 들어 성장주였던 기업이 성숙주로 바뀌면, 매출 성장률보다 현금흐름과 배당에 더 민감해질 수 있다.
따라서 민감도는 고정값이 아니라 계속 갱신되는 값으로 봐야 한다.

## 8. 예측 대상

하나의 기간만 예측하지 말고 여러 기간을 나눠서 본다.

| 예측 대상           | 의미          |
| --------------- | ----------- |
| 발표 후 1일 수익률     | 단기 이벤트 반응   |
| 발표 후 5일 수익률     | 단기 재평가      |
| 발표 후 20일 수익률    | 약 1개월 반응    |
| 발표 후 60일 수익률    | 중기 반응       |
| 다음 재무제표 전까지 수익률 | 분기 단위 투자 성과 |

전략 판단의 중심은 20일과 60일 예상 수익률로 둔다.

## 9. 신호 판단 구조

전략은 우선 Long-only로 시작한다.

매수 후보는 다음 조건을 만족해야 한다.

```text
예상 20일 수익률이 양수
예상 60일 수익률이 양수
재무 품질이 악화되지 않음
부채 부담이 급격히 증가하지 않음
현금흐름이 이익을 따라오고 있음
시장 필터가 위험 구간이 아님
```

제외 조건은 다음과 같다.

```text
영업이익 적자 지속
영업현금흐름 지속 악화
자본총계 감소
부채비율 급등
매출 감소와 재고 증가가 동시에 발생
밸류에이션이 과거 평균 대비 지나치게 높음
데이터 결측이 과도함
```

## 10. 점수화 방향

각 종목은 최종 점수로 정렬한다.

점수는 다음 요소를 반영한다.

```text
예상 20일 수익률
예상 60일 수익률
재무 품질 점수
밸류에이션 점수
시장/섹터 보정
```

예상 수익률이 높더라도 재무 품질이 나쁘거나 밸류에이션이 과열되어 있으면 점수를 낮춘다.

반대로 실적 개선, 현금흐름 개선, ROE 개선, 저평가 상태가 동시에 나타나면 점수를 높인다.

## 11. 포트폴리오 구성 원칙

초기 버전은 단순하게 구성한다.

```text
Long-only
상위 n개 종목 동일비중
종목당 최대 비중 제한
재무제표 이벤트 또는 정해진 주기로 리밸런싱
현금 보유 허용
```

초기 단계에서는 복잡한 비중 최적화보다 전략 아이디어 검증을 우선한다.

고도화 단계에서 다음을 검토한다.

```text
섹터 중립
변동성 조정 비중
Long-short
시장 국면별 비중 조절
팩터별 신뢰도 기반 비중 조절
```

## 12. 리스크 관리

재무제표 기반 전략은 단기 매매 전략이 아니므로 손절을 너무 짧게 잡지 않는다.

기본 원칙은 다음과 같다.

```text
기본 손절은 넓게 설정
익절보다 리밸런싱 중심 청산
예상 수익률이 음수로 바뀌면 청산 검토
다음 재무제표 발표 전후로 재평가
시장 지수가 장기 추세 아래로 내려가면 신규 진입 제한
```

시장 필터는 기본적으로 사용한다.

예시:

```text
미국 주식: SPY 또는 QQQ 기준
한국 주식: KOSPI 또는 KOSDAQ 기준
장기 이동평균 아래에서는 신규 진입 제한
```

## 13. 백테스트 검증 항목

가장 중요한 검증 항목은 미래 데이터 누수 방지다.

필수 확인 사항은 다음과 같다.

```text
재무제표 기준일이 아니라 사용 가능일 이후부터 데이터가 쓰이는가?
미래 분기 재무 데이터를 과거 시점에서 사용하지 않는가?
섹터 평균 계산에 미래 데이터가 섞이지 않는가?
스케일링이나 평균 계산이 전체 기간 기준으로 처리되지 않는가?
예측에 사용된 데이터보다 수익률 타깃 시점이 뒤에 있는가?
결측 데이터가 있어도 전략이 비정상적으로 중단되지 않는가?
```

```text

MDD
CAGR
Sharpe Ratio
승률
평균 보유 기간

거래 로그
```

## 18. 핵심 지시 요약

이 작업은 단순한 재무제표 스크리너가 아니다.

목표는 다음이다.

```text
재무제표 변화와 주가 반응 사이의 종목별 민감도를 학습하고,
새 재무제표가 발표되었을 때 그 변화가 향후 주가에 미칠 가능성을 추정하는 전략을 trading_lab 안에 설계한다.
```

가장 중요한 원칙은 다음 세 가지다.

```text
1. 재무제표 기준일과 사용 가능일을 반드시 분리한다.
2. initialize 구간에서 종목별 민감도를 학습한다.
3. 이후 rolling 방식으로 민감도를 업데이트하면서 예상 수익률을 계산한다.
```

1차 버전은 단순하고 검증 가능한 구조로 시작한다.

```text
Yahoo Finance
미국 주식
Long-only
Ridge Regression
Rolling 12~16분기
20일/60일 예상 수익률
상위 n개 동일비중 포트폴리오
미래 데이터 누수 방지 검증
```

## 19. 핵심 구조 결정 — 단일 종목 파이프라인과의 정합 (★가장 중요)

이 레포의 백테스트 계약은 **1 run = 1 종목**이다.
`StrategyHandler.build_artifacts`는 한 종목의 `forecast`(단일 시계열) ·
`trades`(그 종목의 진입/청산 목록) · `equity`(1.0 기준 단일 누적성장) ·
`metrics`를 돌려준다. 반면 §9~§11의 "top-n 동일비중 포트폴리오 / 섹터 /
리밸런싱"은 **여러 종목을 가로질러** 비중을 정하는 횡단면 로직이라
공통 파이프라인이 호스팅할 수 없다.

따라서 **여러 종목을 병렬로 다루는 2단계 구조**로 확정한다(사용자 확정 2026-06-14).
두 단계 모두 1차 범위이며, "병렬 실행"은 양쪽에 공통으로 얹는다(§28).

| 단계 | 내용 | 결과 단위 | 수용 위치 |
| --- | --- | --- | --- |
| A. 배치 스크리너 | 종목별 민감도 전략을 **종목 리스트에 걸쳐 병렬 run** | 종목별 성과 **비교표** | 단일 종목 핸들러 `fin-sensitivity-v1` + 배치 러너 |
| B. 횡단면 포트폴리오 | 매 리밸런싱마다 전 종목 랭킹→top-n→동일비중 | **단일 포트폴리오 equity 1개** | 포트폴리오 핸들러 `fin-portfolio-v1` (유니버스=종목) |

근거 / 단계 분리 이유:
- A를 먼저 만들면 ⓐ병렬 실행 인프라 ⓑ종목별 민감도 검증 ⓒ횡단면 풀링용
  **패널 데이터**가 자연히 확보되어, 그 위에 B를 얹는 게 깔끔하다.
- A는 §18 "**종목별 민감도**" 지시와 정확히 일치하고, 단일 종목 계약을
  그대로 재사용하므로 계약 테스트(`tests/test_strategy_contract.py`)를 변형 없이 통과한다.
- B는 §27에서 지적한 *단일 종목 Ridge 표본 부족*을 횡단면 풀링으로 완화한다.

**B를 공통 파이프라인에 얹는 방법** — "포트폴리오 전체를 가상의 단일 종목 1개로
래핑": 포트폴리오 NAV/exposure를 `forecast`로, 각 종목 보유구간을 `trade` 하나로,
포트폴리오 누적성장을 `equity`로 매핑한다. 이렇게 하면 B도 **등록된 전략**으로서
공통 대시보드 "결과/비교" 화면에 그대로 뜬다(별도 페이지 신설 금지 — CLAUDE.md 준수).
forecast가 종목별 파형이 아니라 포트폴리오 집계지표(보유종목 수, 평균 pred_ret,
시장필터 상태 등)가 되는 점만 A와 다르다.

## 20. StrategyHandler / StrategyArtifacts 매핑 (A층 구현 계약)

`src/trading_lab/strategies/base.py`의 두 메서드로 구현한다.
`registry._STRATEGIES`에 1줄 등록 + `configs/strategies/fin_sensitivity_v1.json`.

- `load_data(symbol, config, *, csv_path=None, synthetic=False) -> DataFrame`
  - 반환은 **OHLCV 시계열**(공통 `raw`). `config["interval"]`/`config["period"]`
    사용. 재무데이터는 여기서 합치지 말고 §21의 사이드로드로 둔다.
  - `synthetic=True`면 재현 가능한 합성 OHLCV + (내부적으로) 합성 분기재무를
    생성한다. 계약 테스트가 이 경로만 사용하므로 **오프라인에서 반드시 동작**해야 한다.
- `build_artifacts(raw, config, *, symbol, phase, bars_per_year) -> StrategyArtifacts`

산출물 매핑:

| 계약 필드 | A층에서의 의미 |
| --- | --- |
| `forecast` | 일봉 DatetimeIndex + `close` 필수. 보조 컬럼(아래 §24)은 자동 노출됨 |
| `trades` | 분기 이벤트마다 만든 진입/청산 1건씩 (단일 종목 long-only) |
| `equity` | 그 종목 전략의 1.0 기준 누적성장 |
| `metrics` | 최소 `trades,hit_rate,total_return,sharpe,max_drawdown` |
| `extras` | `{"sensitivity_table": DataFrame, "pred_vs_real": DataFrame}` 등 expander용 |

phase 슬라이싱: DI처럼 train/validation/test split을 두고 `phase`로 잘라낸다
(§6 initialize 구간 = train, 매매 평가는 validation 이후).

## 21. 데이터 계층 — 재무데이터 사이드로드 + Point-in-Time

OHLCV는 기존 경로(yfinance/synthetic/csv)로 충분하나, 분기 재무는 별도 수집·정렬이 필요하다.

1. **수집기 신설**: `scripts/fetch_fundamentals.py` (DI의 `scripts/fetch_ohlcv.py` 대응).
   yfinance `quarterly_financials / quarterly_balance_sheet / quarterly_cashflow`로
   §4 팩터를 수집해 `var/fundamentals/<SYMBOL>.parquet`에 저장.
2. **사용 가능일(as-of) 컬럼을 데이터에 박아둔다** (§5의 누수 방지를 데이터 단계에서 강제).
   - 발표일이 있으면 `available_date = 발표일`.
   - 없으면 보수적으로 `available_date = 분기말 + 45일`(분기), `결산말 + 90일`(연간).
   - `build_artifacts`는 일봉 인덱스에 **as-of merge**(merge_asof, `available_date` 기준)로
     재무 팩터를 붙인다 → 각 거래일에는 "그날까지 공개된" 재무만 보인다.
3. **합성 재무 생성기**: 합성 OHLCV와 정렬된 합성 분기재무를 결정적으로 생성하되,
   "팩터 변화 → N일 후 수익률"에 **약한 인과관계를 의도적으로 주입**한다.
   그래야 Ridge가 학습할 신호가 존재하고, 계약 테스트가 의미 있는 trades를 만든다
   (계약 테스트는 trades가 비어도 통과하지만, 회귀 검증용으로 1건 이상이 바람직).

yfinance는 발표일을 신뢰성 있게 주지 않으므로 **45/90일 보수 룰이 1차 기본값**이고,
이 값을 config로 노출해 민감도를 따로 본다.

## 22. 민감도 모델 구체화 (Ridge / rolling)

§7을 구현 가능한 수준으로 고정한다.

```text
피처:   §4 팩터의 "변화량"(전분기 대비 Δ, 표준화). 밸류에이션은 §4의
        과거평균/섹터평균 대비 상대값(z-score)으로.
타깃:   발표 사용가능일 기준 forward 20일 / 60일 누적수익률 (두 모델).
표준화: rolling 윈도우 **안에서만** fit (전체기간 누수 금지 — §13 4번 항목).
학습:   rolling 12~16분기 Ridge. alpha는 config 노출, 분기마다 refit.
인과성: 피처 시점 < 타깃 실현 시점을 코드로 단언(§13 5번). 타깃이 미래로
        삐져나오는 마지막 분기들은 학습에서 제외(forward window 미완성).
출력:   각 분기 발표 시 pred_ret_20d, pred_ret_60d + 팩터별 계수(=민감도).
```

1차는 종목 자기 시계열만으로 Ridge를 적합한다(횡단면 풀링은 2차).
표본이 12분기 미만이면 `insufficient_train_data` 플래그를 metadata에 실어
DI와 동일하게 "학습 부족"을 표면화한다.

## 23. 신호 → trades 변환 (단일 종목 long-only)

§9의 매수/제외 조건을 단일 종목의 진입/청산 이벤트로 환원한다.

```text
진입(long, direction=+1):
  분기 발표 사용가능일 다음 봉(next_open)에서,
    pred_ret_20d > 0 AND pred_ret_60d > 0
    AND §9 제외조건 미해당(적자지속/현금흐름악화/자본감소/부채급등/밸류과열/결측과다)
    AND 시장필터 정상(§12, SPY 장기 MA 위)
청산(exit_reason):
  rebalance   : 다음 분기 재무 발표 도달 (기본 청산)
  horizon     : 60일 보유 한도 도달
  signal_flip : 갱신된 예상수익률이 음전환
  stop_loss   : 넓은 손절(§12)
```

새 `exit_reason` 라벨(`rebalance`,`signal_flip`)은 **`presentation.py`의
`EXIT_REASON_LABELS`에만 추가**한다(공통 인프라 중 라벨 확장은 계약상 허용).
공통 파이프라인 코드 본문은 건드리지 않는다.

## 24. 보조지표 자동노출 활용

`forecast`의 OHLC 외 숫자 컬럼은 대시보드 파형 패널에 **자동 노출**된다
(전략 전용 차트 코드 금지). 다음을 일봉 인덱스에 step-함수(이벤트 시 갱신,
사이는 ffill)로 실어 보낸다.

```text
pred_ret_20d, pred_ret_60d      예상 수익률
quality_score, valuation_z      재무품질 / 밸류에이션 점수(§10)
sens_<factor>                   주요 팩터별 민감도(계수)
```

config의 `dashboard.default_indicators` / `indicator_labels`로 기본 표시·한글
라벨을 지정한다(DI config 참조). §14의 "예측 vs 실제 산점도",
"민감도 변화 그래프"는 `extras`(`pred_vs_real`, `sensitivity_table`)로 넘겨
결과 화면 expander에 띄운다.

## 27. 미해결 / 확인 필요

```text
- §19 결정: 단계적 2단계(A 배치 스크리너 → B 횡단면 포트폴리오)로 확정됨(2026-06-14).

  유의미한 민감도가 안 나오면 B의 횡단면 풀링을 앞당겨야 할 수 있음(본 전략 핵심 실증 리스크).

  섹터상대 비교는 패널이 있는 B에서 활성화한다.
- 유니버스 정의(어느 종목 집합?), 리밸런싱 주기(분기 발표 이벤트 vs 고정 주기),
  top_n·종목당 최대비중은 B config로 노출하되 기본값은 §11을 따른다.
```

## 28. 병렬 실행 아키텍처 (A·B 공통)

"여러 종목 병렬 테스팅"의 실행 계층. 전략 로직과 분리된 **오케스트레이션**이며
공통 인프라(`service.py`/`app.py`)는 건드리지 않는다.

**A · 배치 스크리너 (`scripts/finance_sensitivity/batch.py`)**

```text
입력: 유니버스(심볼 리스트), 전략 id, config 오버라이드
실행: concurrent.futures 로 심볼마다 BacktestService.run(BacktestRequest(symbol=s))
      → 각 run은 기존 단일 종목 계약대로 독립 아티팩트 생성
집계: 각 run의 metrics.json 을 모아 종목별 성과 비교표(DataFrame) 생성
표시: 공통 대시보드의 기존 "비교" 화면이 run들을 그대로 보여줌(신규 페이지 불필요)
```

병렬화 주의:
- run끼리 독립이라 종목 단위 병렬은 안전. **그러나 `RunStore`는 SQLite** —
  다중 프로세스 동시 쓰기 경합 가능. 1차는 (ⓐ)yfinance 수집/모델만 워커에서
  병렬하고 DB 쓰기는 메인에서 직렬화, 또는 (ⓑ)WAL 모드 + 짧은 트랜잭션으로 처리.
  과한 병렬보다 **수집(IO-bound) 위주 스레드풀**이 효과 대비 안전.
- 재현성: 워커별 시드/캐시 키를 심볼에 고정. 실패 종목은 건너뛰고 비교표에
  `status=failed`로 남겨 전체 배치가 중단되지 않게 한다(§13 "결측에도 비정상 중단 금지").
